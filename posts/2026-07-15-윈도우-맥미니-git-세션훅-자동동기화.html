맥미니에서 물어본 걸 여기서도 보려면? — 이 질문 하나가 자동 동기화 시스템이 됐다

"맥미니에서 너한테 물었을 때 답한 게 있는데, 그 세션을 여기서도 확인할 수 있어?" 라는 질문에서 시작됐다. 확인해보니 세션 자체는 기기별 로컬 저장소에 있어서 안 됐다 — 다만 볼트는 구글드라이브+깃허브로 두 기기가 공유하고 있으니, "그럼 너는 답하기 전에 깃에서 풀부터 하고 답하는 거야?"라는 다음 질문이 훨씬 중요했다. 답은 "아니오"였다.

1. 없는 걸 있는 척하고 있었다

세션 시작 훅(session_start.py)은 할일·일정 동기화, 최근 작업 맥락 복원, 신규 파일 스캔은 하고 있었지만 git pull은 아예 하지 않았다. 즉 맥미니에서 뭘 커밋·푸시해놔도, 윈도우 세션은 그 내용을 전혀 모른 채로 옛날 파일 기준으로 대답하고 있었을 수 있다는 뜻이다. 반대 방향(윈도우→맥미니)도 마찬가지였다.

2. 대칭 구조로 설계

원칙은 단순했다 — fetch로 원격 상태 먼저 확인한 다음에만 움직인다. scratch/git_sync.py에 두 함수를 만들었다.

  • pull_if_behind(): 세션 시작 시. 원격이 앞서 있으면, 로컬 미커밋 변경을 stash로 잠깐 치워두고 fast-forward pull한 뒤 다시 복원한다.
  • push_if_needed(): 세션 종료 시. 먼저 pull_if_behind()로 맞추고, 로컬 변경을 "vault backup: 시각"으로 커밋한 뒤 push한다.

.claude/settings.json에 SessionEnd 훅을 추가해 push_if_needed()를 세션 종료 시 자동 실행하도록 연결했다. 세션 시작은 이미 있던 훅에 pull_if_behind()만 끼워 넣었다.

3. 실전 테스트에서 바로 사고가 났다

만들자마자 실제 상태(원격 2커밋 앞섬 + 로컬 미커밋 변경 있음)로 테스트했는데, git stash push -u가 exit code 1을 냈다. 구글드라이브 동기화 폴더 안의 빈 디렉토리를 정리하려다 권한 오류가 난 것뿐이었는데, 코드는 이걸 "stash 실패"로 오판해서 그냥 멈춰버렸다. 그런데 실제로는 stash 자체는 이미 저장돼 있었다 — 즉 되돌리지 않은 채 방치할 뻔했다.

다행히 stash 목록을 직접 열어보니 지운 줄 알았던 파일들(빠른메모 노트 2개)이 그 안에 고스란히 들어 있었다. git stash apply로 복원하고, 이후엔 원인을 근본적으로 고쳤다 — -u(untracked 포함) 플래그를 아예 빼서 애초에 그 권한 오류가 날 지점을 없애고, exit code 대신 실제 working tree 상태로 성공 여부를 재확인하도록 바꿨다.

4. 맥미니가 먼저 답을 보내왔다

이 파일들이 git으로 추적되는 덕분에, 맥미니 쪽 초록이 pull을 받자마자 경로 하드코딩을 없애 크로스플랫폼으로 고쳐서 다시 커밋해 보냈다. 굳이 맥·윈도우용을 따로 만들 필요가 없어진 셈이다.

마무리

이제 두 기기 모두 세션을 열고 닫을 때마다 "원격에 뭐가 있는지 먼저 확인 → 있으면 안전하게 반영"이 자동으로 돈다. 다만 이번에 배운 건 하나 더 있다 — subprocess exit code만 믿고 성공·실패를 판단하면 안 된다는 것. 진짜 상태(working tree가 실제로 깨끗해졌는가)를 다시 확인하는 한 줄이 없었다면, 아무 것도 모른 채 로컬 파일 두 개를 조용히 잃어버릴 뻔했다.
