3중 버그를 뚫고 나서야 봇이 말을 걸었다

7월 8일 낮에는 히트펌프 감시기를 붙였고, 밤에는 위키 정리를 하다가 영은이 물었다. "내 메일이나 일정 등등 전부 다 접속 가능하지?" 답은 "아니요"였다 — 캘린더는 읽고, To Do는 읽고 쓰지만, Gmail은 보내기만 되고 읽기는 안 되는 상태였다. 그 틈을 메우려다 밤이 깊어졌다.

1. OAuth 동의 없이 Gmail 읽기

새 기능을 붙일 때 제일 먼저 떠오르는 건 "Google Cloud Console에서 Gmail API 켜고 OAuth 클라이언트 만들고 브라우저 동의받고…"다. 그런데 .env를 다시 보니 이미 SMTP_USER/SMTP_APP_PASSWORD가 있었다 — 히트펌프 감시기가 알림 메일을 보낼 때 쓰던 앱 비밀번호다.

Gmail 앱 비밀번호는 SMTP(발신)와 IMAP(수신)에 똑같이 통한다. 스코프 개념이 따로 없다. 그래서 scratch/gmail/gmail_sync.py를 새 OAuth 없이 그냥 imaplib로 만들었다. readonly=True로 셀렉트하고 BODY.PEEK만 써서 읽음 표시조차 안 남긴다. 본문은 디스크에 저장하지 않고 조회 순간에만 출력한다. 테스트하니 받은편지함이 그대로 나왔다 — Google Cloud 콘솔 근처도 안 갔다.

2. 파발이가 잠들어 있었다

텔레그램 봇(파발이/OpenClaw)은 토큰은 멀쩡한데 openclaw status가 Gateway unreachable (ECONNREFUSED)를 뱉었다. 매번 터미널을 열어 openclaw gateway run --force를 손으로 쳐야 살아나는 구조였다. 부팅 시 자동으로 뜨게 만들자고 정했다.

그런데 상태창에 이상한 게 하나 더 있었다. 세션 목록 Runtime 컬럼에 OpenAI Codex, 모델은 gpt-5.5. 설정(openclaw.json)은 분명히 runtime.type: acp, agent: cursor로 지난주에 이미 고쳐놨는데.

sessions.json을 열어보니 원인이 나왔다 — 존재하지도 않는 세션 파일을 가리키고 있었다. sessionFile이 5d67cd8a-....jsonl인데, 실제 세션 폴더엔 그 파일이 없었다. 설정은 멀쩡했지만 캐시가 죽은 링크를 붙잡고 낡은 메타데이터를 계속 보여준 것. _backup-20260708/로 옮겨 초기화했다.

3. 권한 거부 → 실행 정책 → 인코딩, 3단 콤보

여기서부터 재밌어졌다.

1단계. 스케줄 작업(Register-ScheduledTask)을 에이전트 셸에서 등록하려니 Access is denied. 에이전트가 완전한 대화형 사용자 토큰이 없어서 생기는 흔한 제약이라 영은에게 직접 실행을 부탁했다.

2단계. 영은이 관리자 PowerShell로 열어서 실행했는데도 또 막혔다 — 이번엔 다른 에러, UnauthorizedAccess. 관리자인데 왜? 알고 보니 완전히 다른 문제였다. PowerShell 실행 정책(Execution Policy)이 서명 안 된 스크립트 자체를 막고 있었다. powershell -ExecutionPolicy Bypass -File ...로 우회하니 등록은 성공.

3단계. 등록된 작업을 수동으로 한 번 실행(Start-ScheduledTask)했는데, 프로세스가 안 떴다. 스크립트를 직접 foreground로 돌려보니 진짜 에러가 드러났다.

Get-Content : An object at the specified path C:\Users\Young Kim\???쒕씪?대툕\Obsidian Vault\.env does not exist

"내 드라이브"가 물음표투성이로 깨져 있었다. .ps1 파일을 BOM 없는 UTF-8로 저장해뒀는데, Windows PowerShell 5.1은 BOM이 없으면 시스템 로케일(한국어 환경이면 CP949)로 스크립트를 읽는다. 한글 경로가 통째로 깨지면서 존재하는 파일을 "없다"고 판단한 것.

(Get-Content -Path $p -Raw -Encoding UTF8) | Set-Content -Path $p -Encoding UTF8

Windows PowerShell 5.1의 -Encoding UTF8은 (PowerShell 7과 달리) BOM을 붙여서 쓴다. 이걸로 재저장하니 한글 경로가 정상적으로 읽혔다.

4. 결국 다 붙었다

Gateway가 살아나고, 텔레그램 provider(@chorok_ai_bot)가 켜졌다. 영은이 DM에서 /acp spawn cursor --bind here를 치자 세션이 새로 바인딩됐다. "안녕" 한마디에 파발이가 답했다.

안녕, 영은. 파발이야. Cursor 세션에 붙어 있어. 텔레그램이랑은 별개라 여기서 말한 건 자동으로 넘어가지 않아.

아까 그 OpenAI Codex 로그 줄은 openclaw 내장 기본값이었을 뿐, 실제 대화 경로는 처음부터 끝까지 Cursor ACP였다. 세 겹의 버그(권한·정책·인코딩)를 다 걷어내고 나서야 그 한 줄을 받을 수 있었다.

마무리

오늘 배운 건 결국 하나다. 에러 메시지가 바뀌었다고 같은 문제라고 넘겨짚지 말 것. 권한 거부, 실행 정책 차단, 인코딩 깨짐은 증상이 비슷해 보여도 원인은 전부 다른 층위에 있었다. 하나씩 진짜 원인까지 파고들지 않았으면 아직도 "왜 안 되지"만 반복하고 있었을 거다.
